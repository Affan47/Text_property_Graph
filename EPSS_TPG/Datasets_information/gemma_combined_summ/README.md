## 1. Structural relationship to `gpt_combined_summ`

These two CSVs are the **same underlying CVE corpus** with different LLM-generated summary text. Everything except the summary content is byte-identical.

| Metric | gpt_combined | gemma_combined | Δ |
|---|---:|---:|---|
| Rows | 9,218 | 9,218 | — |
| Columns | 27 | 27 | — |

| EPSS mean | 0.1067 | 0.1067 | — |

**Implication:** every structural finding from [gpt_combined_summ](../gpt_combined_summ/README.md) — the multi-row-per-CVE leakage, the same 40.3 % base-CVE overlap on random splits, the same `code_available`/`source_count` proxies, the same `epss_status='enriched'` 33 % imputed slice — applies here unchanged. The **only varying surface is the `summ_all_sources` text**, generated by Gemma instead of GPT-OSS.

## 2. Features (27 columns)

Identical schema to `gpt_combined_summ`. See its [features documentation](../gpt_combined_summ/README.md#1-features-27-columns) for the per-column description. The fields that differ in **content** (not schema) are:

| Column | gpt_combined origin | gemma_combined origin |
|---|---|---|
| `summ_all_sources` | GPT-OSS LLM summary across all source links | **Gemma LLM summary across all source links** |
| `summ_github_urls` | GPT-OSS LLM summary of GitHub-URL sources | **Gemma LLM summary of GitHub-URL sources** |

Both are renamed by `epss/prepare_dataset.py` to `summary` (only `summ_all_sources` is mapped — `summ_github_urls` stays as an extra column).

## 4. Critical findings

Inheriting from [gpt_combined_summ §4](../gpt_combined_summ/README.md#4-critical-findings) — same dataset structure means the same four flagged issues apply:

| # | Finding | Same in this dataset? |
|---|---|---|
| 4.1 | Multi-row-per-CVE causes 40 % base-CVE overlap on random splits | ✅ identical |
| 4.2 | LLM `summary` may leak target via exploitation phrasing | ⚠ **content differs** — Gemma's wording may leak more, less, or similarly |
| 4.3 | `delta_days_*` 80 % null | ✅ identical |
| 4.4 | `-N` suffix breaks external joins | ✅ identical |

### What's specific to this dataset

**Finding 4.2 is the natural A/B for the gpt vs gemma comparison.** Holding the row structure, EPSS labels, CVSS, and tabular features constant across the two datasets, the only thing that varies is the summary text. Comparing PR-AUC on Run A (baseline, summary present) and Run D (summary dropped) across both datasets directly attributes any metric difference to the summarizer.

## 5. Experiment plan

Repeating the **same 8-run ablation matrix** that was executed on `gpt_combined_summ` ([results](../gpt_combined_summ/ablation_results.md) §1 + §8.3). Each run uses identical model architecture, hyperparameters, and split logic — only the prepare-time flags differ.

| Run | Flags | Tests |
|---|---|---|
| A | none | Baseline |
| B | `--dedupe-by-base-cve` | Multi-row leakage |
| C | `--dedupe-by-base-cve --drop-summary` | Multi-row + LLM summary text |
| D | `--drop-summary` | Gemma summary contribution |
| E | `--drop-tabular-leaks` | `code_available` + `source_count` |
| F | `--filter-original-epss` | Imputed labels |
| G | `--filter-original-epss --drop-tabular-leaks` | E ∧ F |
| H | all four flags stacked | Max-clean realistic baseline |

After all 8 runs complete, `ablation_results.md` (see below) will be populated with the same comparison and verdict structure as the GPT report.
