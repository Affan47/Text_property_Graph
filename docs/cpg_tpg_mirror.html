The big idea

Four dimensions, same structure

A CPG encodes code across four layers: syntax tree (AST), execution order (CFG), data flow (DFG), and control dependence (CDG). Natural language text has exact analogs for each.

Syntactic structure — layer 1

CodeAST — parse tree (parent → child)
TextDEP — dependency tree (head → dependent)
WhyBoth decompose structure into typed parent-child relationships

Sequential flow — layer 2

CodeCFG — statement execution order
TextNEXT_TOKEN / NEXT_SENT / NEXT_PARA
WhyBoth define temporal/sequential flow through the content

Data / entity flow — layer 3

CodeREACHING_DEF — variable propagation
TextCOREF — entity coreference chains
WhyBoth track "this thing" from definition to later uses

Discourse / control — layer 4

CodeCDG — control dependence
TextRST_RELATION — discourse dependence
WhyBoth capture "because of X, then Y" relationships

Key insight: DEP_LABEL = the edge type within the syntax layer

In Joern's CPG, all syntax edges share the type AST — the specific relationship is inferred from the child's node type. In the TPG, all syntax edges share the type DEP — but each edge carries an explicit DEP_LABEL (nsubj, det, prep, etc.) encoding the exact syntactic role. The TPG's syntax layer is richer per-edge than Joern's.

Practical impact for GNNs

When exported to PyG format, each unique DEP_LABEL becomes a distinct edge type in data.edge_type. A relation-aware GNN (RGAT/RGCN — same as SemVul) can learn separate message-passing weights for nsubj vs dobj vs prep, just as it learns different weights for AST vs CFG vs REACHING_DEF in the CPG.
