Motivation

Why generic NLP fails on security text

spaCy's NER model was trained on news and Wikipedia. It has no concept of CVEs, CWEs, vulnerability types, or dangerous functions. When processing a CVE advisory, it misclassifies or misses nearly every security-relevant entity.

spaCy generic NER (Level 1)

"CVE-2024-1234"CARDINAL or missed
"Apache HTTP Server"ORG (wrong)
"2.4.51"CARDINAL (wrong)
"buffer overflow"just two words
"strcpy"gibberish to NER
"CWE-120"missed entirely
→

"Apache HTTP Server"SOFTWARE (set)
"2.4.51"VERSION (regex)
"buffer overflow"VULN_TYPE (dict)
"strcpy"CODE_ELEMENT (regex)
"CWE-120"CWE_ID (regex)

Architecture

The Joern frontend analogy

Joern doesn't use one parser for every language — each language has its own frontend. All frontends output the same CPG schema with language-specific extensions. The TPG mirrors this exactly: domain-specific frontends extend the generic NLP frontend.

Joern — multiple language frontends

C/C++ frontend→CDT parser → C-specific AST nodes
Java frontend→Soot/JavaParser → Java-specific nodes
Python frontend→Custom parser → Python-specific nodes
Binary frontend→Ghidra → Binary-specific nodes
All output the same CPG schema (METHOD, CALL, IDENTIFIER, AST, CFG, REACHING_DEF) with language-specific properties.

TPG — multiple domain frontends

SpacyFrontend→Generic NLP → Base text nodes (Level 1)
SecurityFrontend→Regex + keywords → Security nodes (Level 2)
MedicalFrontend→BioBERT NER → Medical nodes (future)
LegalFrontend→Legal NER → Legal nodes (future)
All output the same TPG schema (DOCUMENT, SENTENCE, TOKEN, ENTITY, DEP, COREF, RST_RELATION) with domain-specific properties.

Implementation

SecurityFrontend 3-step pipeline

SecurityFrontend inherits from SpacyFrontend. When parse() is called, it runs the full generic NLP first, then overlays security-specific extraction, then creates domain relationship edges.

Step 1 — SpacyFrontend.parse() (inherited)

a
Tokenization + POS tagging
Break text into tokens, assign parts of speech
→ 50+ TOKEN nodes with POS, LEMMA properties
b
Dependency parse + generic NER
Syntax tree + named entities ("Apache" → ORG, wrong but still created)
→ DEP edges, ENTITY nodes, SENTENCE/PARAGRAPH/DOCUMENT structure
c
Sequential flow + containment
NEXT_TOKEN, NEXT_SENT, NEXT_PARA, CONTAINS edges
→ Full Level 1 TPG (all base nodes + edges)

Domain knowledge

The 6 extraction categories

All security-domain knowledge is hardcoded as module-level constants — regex patterns for structured identifiers, dictionaries for normalized terminology. These are not ML models — they are deterministic rules.

Regex patterns — structured IDs

CVE IDs, CWE IDs, version numbers, CVSS scores, and severity levels always follow predictable formats.

METHOD: re.compile()
  • CVE-\d{4}-\d{4,} → CVE-2024-1234
  • CWE-\d{1,4} → CWE-120
  • \d+\.\d+(?:\.\d+)* → 2.4.51
  • (critical|high|medium|low) → severity levels
  • strcpy|strcat|gets|eval|... → dangerous functions

Software names — 35 known products

Case-insensitive substring match against a set of known product names. Extensible.

METHOD: set lookup
  • apache, nginx, iis, tomcat web servers
  • openssl, openssh crypto/net
  • mysql, postgresql, redis databases
  • docker, kubernetes, jenkins devops
  • log4j, struts, jackson notorious libs

Attack vectors — 9 types

Maps attack vector phrases to CVSS-aligned normalized types. Critical for severity scoring.

METHOD: dictionary lookup
  • "remote attacker" → remote
  • "user-supplied input" → user_input
  • "specially crafted" → crafted_input
  • "local" → local
  • "adjacent" → adjacent_network

Impact types — 18 types

Normalizes varied descriptions to canonical impact categories. "arbitrary code execution" and "remote code execution" both map to rce.

METHOD: dictionary lookup
  • "arbitrary code execution" → rce
  • "denial of service" → dos
  • "privilege escalation" → privesc
  • "use after free" → uaf
  • "sql injection" → sqli

Vulnerability types — 22 types

OWASP/CWE-standard vulnerability classes. Maps natural language descriptions to normalized CWE-aligned categories.

METHOD: dictionary lookup
  • "buffer overflow" → buffer_overflow
  • "use-after-free" → use_after_free
  • "cross-site scripting" → xss
  • "insecure deserialization" → deserialization
  • "hardcoded credentials" → hardcoded_creds

Remediation — 6 regex patterns

Captures fix/patch/upgrade guidance from advisories. More variation than IDs, so regex is used.

METHOD: regex patterns
  • upgrad\w* to (\S+) → upgrade
  • patch\w* by applying → patch
  • workaround → workaround
  • mitigat\w* → mitigation
  • disabl\w* \w+ → disable_feature

Node schema

Security entity types

All security entities are ENTITY nodes (same TPG node type) with additional properties: ENTITY_TYPE (security category), DOMAIN_TYPE (normalized subcategory), CONFIDENCE (1.0 for regex, 0.8–0.95 for keyword), and SOURCE: "security_frontend".

Worked example

CVE advisory walkthrough

The input text below is processed through the SecurityFrontend. Entities are annotated with their extraction color: CVE, CWE, SOFTWARE, VERSION, VULN_TYPE, CODE_ELEMENT, ATTACK_VECTOR.

CVE advisory input
CVE-2024-1234: A buffer overflow vulnerability (CWE-120) has been discovered in Apache HTTP Server version 2.4.51. The flaw exists in the mod_ssl module where user-supplied input is copied to a fixed-size buffer using strcpy without bounds checking. Users should upgrade to version 2.4.52.

Pipeline trace:

  • 50+ TOKEN nodes
  • DEP edges (syntax tree)
  • NEXT_TOKEN/SENT/PARA edges

Step 2 output (security)

  • CVE-2024-1234 conf: 1.0
  • CWE-120 conf: 1.0
  • Apache HTTP Server conf: 0.9
  • 2.4.51, 2.4.52 conf: 1.0
  • buffer overflow conf: 0.9
  • strcpy conf: 1.0
  • user-supplied input conf: 0.85
  • upgrade to 2.4.52 conf: 0.8

Step 3 output (edges)

  • CVE → Apache AFFECTS
  • CVE → CWE-120 CLASSIFIED_AS
  • Apache → 2.4.51 HAS_VERSION
  • buf overflow → strcpy USES_FUNCTION
